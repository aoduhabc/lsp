# “传统形式化验证”的难点
---

## 1.1 本质难点：智能合约的形式化验证首先是“规格（Specification）与假设（Assumptions）工程”，这是一个建模问题

形式化验证的核心并不是“求解器能不能解”，而是能否把要证明的命题表达清楚：**在给定语义与环境假设下，合约实现满足某个安全性质**。形式化地说，一个可交付的验证结论应当被表述为：

> 在语义模型 **S**（例如 EVM 的回滚、调用、异常传播等语义）与环境/依赖假设 **A**（例如外部合约摘要、预言机/价格假设、可达性约束等）成立的前提下，证明合约实现 **C** 满足性质 **P**。  
> 记作：**(S, A) ⊢ C ⊨ P**

由此可见，“第一道坎”不是 Z3/SMT 的性能，而是 **P 与 A 是否被准确、完备且可审计地表达出来**：  
- **P（系统行为级性质）**：应尽量描述系统应当遵守的总体行为与安全边界，而非把实现细节“翻译一遍”。因为复述实现会把 bug 同时“写进规格”，导致验证失效。  
- **A（假设/边界/TCB）**：必须把性质成立所依赖的外部条件显式写出来，否则验证结论无法被解释、复现与迁移。

> 说明：工程上常使用 summaries / harness / 差分化抽象来降低复杂度、提高可解性，但这些属于“规模化手段”，不是逻辑必需前提；一旦引入抽象，它们必须被纳入 A/TCB 作为公开假设，否则结论不可审计。

## 1.2 合约与语义对齐难：语义模型 S 一旦偏差，验证结论整体不 sound

传统形式化验证的 soundness 取决于语义模型 **S** 是否正确表达目标执行环境。对智能合约而言，EVM 的诸多细节（例如 revert/异常传播、delegatecall/callcode 的上下文语义、gas 与异常路径等）一旦建模出现偏差，会引入 **unsoundness** 风险：  
- 你可能“证明通过”，但该结论只在错误语义 S’ 下成立，而不在真实 EVM 下成立。  
- 因此必须把语义相关的可信边界（TCB）显式披露，并尽量压缩：哪些语义由工具严格保证，哪些语义通过近似/摘要处理。

**修复要求（写作与工程同时约束）：**
1. 将语义可信边界单独列为 **TCB(S)**：明确哪些 EVM 语义点被精确覆盖，哪些未覆盖或被近似。  
2. 将抽象/摘要/近似单独列为 **TCB(A)**：避免把“语义正确性问题”与“规模化近似问题”混为一谈。  
3. 对每条关键性质 P，明确它对语义细节的敏感点（例如是否依赖回滚、是否依赖跨合约上下文一致性）。

## 1.3 建模中的“预言机/环境缺口”问题：外部依赖未知会把 A 变成隐含假设，从而削弱结论

传统形式化验证在 DeFi/多合约系统中会遭遇典型的“oracle gap”：**系统真实行为依赖外部组件与外部数据，但验证时这些组件/数据往往缺失或被近似**。如果不把这些缺口显式化，它们会以“隐含假设”的形式潜入证明，使结论无法转化为可靠安全结论。

将“预言机问题”拆为两类，并分别给出修复路径：

### (a) 外部组件缺失 / 依赖缺口（Environment Gap）
在 multi-contract 场景中，外部合约实现未知、fallback 行为、dispatcher summaries 的限制，会把跨合约调用的关键行为变成隐含假设。这意味着“只验证单合约”通常不足：你必须处理跨合约边界与外部依赖，否则 P 的成立范围不清晰。

**修复要求：**
- 明确系统边界：哪些合约属于验证对象，哪些属于环境。  
- 对环境合约给出最小可审计摘要（summary/spec），或明确采用的建模近似（并写入 TCB(A)）。  
- 对 fallback/dispatcher 等机制给出显式约束：哪些路径被覆盖，哪些路径被排除，排除原因是什么。

### (b) 经济预言机 / 市场假设（Economic Oracle Assumptions）
DeFi 的价格变化、健康度、流动性突变等属于环境输入的非平稳行为。此时性质 P 往往不是“纯代码性质”，而是“代码 + 经济环境”的组合性质。如果不把这些假设显式写入 **A/TCB**（例如价格变化边界、预言机更新频率、攻击者能力模型），即便证明为真，也难以转化为可靠风险结论。

**修复要求：**
- 为每条性质 P 配套写出 **Assumptions(A)**：例如价格源可信条件、操纵窗口、最大滑点、可调用次数/顺序等。  
- 将“可验证内容”与“需外部承诺内容”分离：把无法在当前模型内证明的部分明确标注为外部前提，而不是默默忽略。
